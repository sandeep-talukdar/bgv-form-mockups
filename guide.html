Spec Breakdown

  1. 1
    Sidebar active item
    "Identity" is highlighted with a blue left border and light blue background. Shows 2 grey status dots indicating PAN and Aadhaar are both incomplete.
    Key states: Active section has blue left border + bg. Completed sections show green check. Upcoming sections are default grey.
    Design tokens: Border color #1e40af, Background #f5f7ff
  2. 2
    Progress bar
    Completed items show a green 140px progress bar with a green check icon. "Basic details" is marked complete here. Each sidebar item can transition from incomplete (grey) to complete (green bar + check).
    Design tokens: Progress bar width 140px, green fill for completed state
  3. 3
    Section title card
    "Identity" section header displaying 2 inline status dots representing PAN and Aadhaar sub-sections. Dots are grey when incomplete, green when done.
    Design tokens: Card height 72px, Border #d9d9d9, Border radius 12px
  4. 4
    Upload zone
    Dashed border area containing the DigiLocker button, an "or" text divider, and a manual upload icon. This is the primary document upload interaction zone.
    Key states: Default (dashed border), Drag-over (highlighted border), Uploading (progress indicator)
    Design tokens: Border 1.5px dashed rgba(8,40,204,0.4), Height 140px. DigiLocker button: 231x44px, Background #6334fa
  5. 5
    Document card
    "PAN Card" uploaded document card. Shows uploaded file preview. When selected/active, the border turns blue to indicate current focus.
    Key states: Default (grey border), Selected (blue border), Uploaded (shows file details)
    Design tokens: Size 233x160px, Header bg #f1f5fc, Default border #e5e7eb, Selected border #0828cc
  6. 6
    "I don't have this document" link
    Blue underlined text link. Clicking this triggers the Exit Flow modal (Scenarios 8-9), allowing the candidate to provide a reason and upload alternate documents.
    Key states: Default (blue underlined), Hover (darker blue)
  7. 7
    Proceed button
    Located INSIDE the PAN subsection card. Disabled (low opacity) because not all sub-sections are complete. Only becomes active when the current sub-section's requirements are met.
    Key states: Disabled (opacity 0.4, not clickable), Enabled (full opacity, clickable)
    Design tokens: Size 168x44px, Background #0828cc, Border radius 8px

Triggered by: Candidate completes Identity step
What happens next: Complete education sub-sections, then move to Address

Spec Breakdown

  1. 1
    "I don't have this education" checkbox
    Positioned on the right side of the subsection header. Allows the candidate to skip an entire education level (e.g., if they don't have a Post-graduation). When checked, the entire sub-section collapses and is marked as "Not applicable".
    Key states: Unchecked (default — sub-section expanded), Checked (sub-section collapses, skipped)
  2. 2
    University/school name dropdown
    Required field. A searchable dropdown populated from API. The candidate types to search and selects their institution from the results.
    Key states: Empty (placeholder text), Searching (loading spinner), Populated (selected value displayed)
  3. 3
    Passing year dropdown
    Required field. Year selection dropdown for when the candidate completed this education level.
  4. 4
    Preferred doc label + Mandatory badge
    Shows which document is preferred for this education level. When configured by the company admin, a red "Mandatory" badge appears next to the label, indicating this specific document must be uploaded.
    Key states: Preferred only (no badge — any document accepted), Preferred + Mandatory (red badge — this specific document required). Driven by BGV Form Config API.
  5. 5
    Upload zone
    Same pattern as Identity. Contains DigiLocker button and manual upload option for education documents.
  6. 6
    "I don't have preferred documents" link
    Blue underlined text link. Clicking triggers the Exit Flow modal (Scenarios 8-9) where the candidate provides a reason and can upload alternate documents.
  7. 7
    DigiLocker action buttons
    Two side-by-side buttons: "Proceed without DigiLocker" (outlined/ghost style) and "Fetch from DigiLocker" (filled green). These appear specifically in the Twelfth and Tenth sub-sections where DigiLocker integration is supported.
    Design tokens: "Fetch from DigiLocker" background #0f973d
  8. 8
    DigiLocker tooltip
    Dark tooltip explaining the benefits of using DigiLocker for document retrieval. Appears on hover or focus near the DigiLocker buttons. Provides helpful context about why DigiLocker is recommended.

What happens next: Upload remaining mandatory docs, then Proceed becomes enabled (Scenario 4)

Spec Breakdown

  1. 1
    Employment type dropdown
    "Full time" selected. This is a required field that determines the type of employment being verified. Options typically include Full time, Part time, Contract, Internship.
  2. 2
    Preferred doc label
    "Preferred: Experience Letter" with a red "Mandatory" badge. Indicates that this specific document is the company admin's preferred proof of employment and is required.
  3. 3
    Uploaded file row
    Replaces the upload zone after the hero document is uploaded. Shows file name, file size, upload timestamp, a green check icon confirming success, and a delete button to remove the upload.
    Key states: Uploading (progress bar), Uploaded (file details + green check + delete), Error (red border + retry)
  4. 4
    "Preferred document uploaded" success
    Green check icon with a success message confirming the preferred/hero document has been uploaded. This visual confirmation appears after the hero document upload completes.
  5. 5
    Mandatory documents group (red)
    A visually distinct group with a red border containing remaining mandatory documents. The header shows "Other mandatory documents" with a count badge indicating how many are left. Each document inside has a Pending or Uploaded status pill.
    Design tokens: Border #fca5a5 (red-300). Transitions to green when all mandatory docs are uploaded (see Scenario 4).
  6. 6
    Upload zone for remaining doc
    Inside the mandatory group. Same DigiLocker + manual upload pattern as other upload zones. Allows uploading the next required document (e.g., Relieving Letter).
  7. 7
    Optional documents list
    Listed below the mandatory group. Each optional document has a "+ Upload" button. These documents never block the Proceed button — they are nice-to-have but not required.
  8. 8
    Proceed button (disabled)
    Located inside the subsection card. Disabled because Relieving Letter (a mandatory document) is still Pending. The button only becomes enabled when all mandatory documents are uploaded.
    Key states: Disabled (opacity 0.4) — at least one mandatory doc is Pending. Enabled (full opacity) — all mandatory docs uploaded.

Employment: All Mandatory Done

All mandatory documents for Employment 1 have been uploaded. The mandatory group turns green, and the Proceed button becomes active.

Triggered by: All mandatory documents uploaded for Employment 1
What happens next: Click Proceed to move to Employment 2 or next section
1
2
3
4

Spec Breakdown

  1. 1
    Mandatory group (green)
    The mandatory documents group transitions from red to green. Border turns green, header background turns light green. Title changes to "All mandatory documents uploaded" with a green check icon and green count badge.
    Design tokens: Border #86efac (green-300), Header bg #f0fdf4 (green-50). Transition from red state: border #fca5a5 to #86efac
  2. 2
    "Uploaded" status pill
    Green pill that replaces the grey "Pending" pill when a document is uploaded. Shows the document is complete and verified.

  3. 3
    Proceed button (ENABLED)
    Full opacity and clickable. All mandatory documents have been uploaded, so the candidate can now proceed to the next employment sub-section or the next section.
    Key states: Enabled (full opacity, cursor: pointer). Clicking advances to Employment 2 if it exists, or the next form section.
  4. 4
    Sidebar status dot
    The first dot (Employment 1) is now a green check, indicating completion. The second dot (Employment 2) remains grey, indicating it is still pending.
    Key states: Grey dot (incomplete), Green check (complete)

Scenario 6

Identity: Mandatory & Optional

Company admin has configured PAN as mandatory, and Aadhaar and Driving License as optional. The Proceed button behavior differs for mandatory vs optional sub-sections.

Triggered by: Company admin configures PAN as mandatory, Aadhaar and DL as optional
What happens next: Upload PAN (required), optionally upload others
1
2
3
4
5

Spec Breakdown

  1. 1
    Mandatory badge on PAN
    Red "Mandatory" badge displayed in the PAN subsection header. This document MUST be uploaded before the candidate can proceed past Identity.
    Visual: Red badge with white text. Positioned inline with the subsection title.
  2. 2
    Optional badge on Aadhaar
    Grey "Optional" badge displayed in the Aadhaar subsection header. The candidate can proceed even without uploading this document.
    Visual: Grey badge. The Proceed button for this sub-section is enabled regardless of upload status.
  3. 3
    Proceed enabled (optional section)
    Aadhaar's Proceed button is at full opacity and clickable because the section is optional. The candidate can skip this document and move on.
  4. 4
    Collapsed card (Driving License)
    The Driving License card is shown in a collapsed state with reduced opacity. Only the header is visible — no upload zone or content is shown. The candidate hasn't opened/expanded it yet.
    Design tokens: Opacity 0.6 when collapsed. Full opacity when expanded. Click on header to expand.
  5. 5
    Proceed disabled (mandatory section)
    PAN's Proceed button is disabled (opacity 0.4) because the mandatory document has not been uploaded yet. The candidate must upload PAN before they can complete this sub-section.
    Key behavior: Mandatory sub-sections block Proceed until the document is uploaded. Optional sub-sections allow Proceed at any time.
Scenario 7

Address: Permanent & Current

The candidate is on the Address section with sub-sections for Permanent and Current addresses. A digital verification warning banner is shown. A checkbox allows copying permanent address to current.

Triggered by: Candidate completes Education step
What happens next: Upload address proof, fill address details
1
2
3
4
5
6

Spec Breakdown

  1. 1
    Digital verification warning
    Yellow informational banner stating "This address will be verified digitally..." Provides context to the candidate about the verification method. This banner is not dismissible.
    Visual: Yellow/amber background, informational icon. Always visible — no close button.
  2. 2
    Acceptable documents list
    Lists the documents accepted for address proof: "Aadhaar (front and back), Driving License..." with a "+13 more" expandable link that reveals the full list of accepted documents when clicked.
    Key states: Collapsed (shows first few + "+N more" link), Expanded (shows full list)
  3. 3
    Upload zone
    Same DigiLocker + manual upload pattern. Allows the candidate to upload their address proof document.
  4. 4
    "I don't have preferred documents" link
    Blue underlined link. Triggers the Exit Flow modal (Scenarios 8-9) for the address section, allowing the candidate to explain why they cannot provide the preferred address proof.
  5. 5
    Proceed button
    Inside the Permanent address card. Disabled until the address proof document has been uploaded.
  6. 6
    "Current address is same as permanent" checkbox
    Positioned on the right side of the Current address header. When checked, auto-copies all permanent address details (address fields and uploaded documents) to the Current address sub-section, saving the candidate from re-entering the same information.
    Key behavior: Checking copies all fields and marks Current address as complete. Unchecking clears the copied data and requires manual entry.
Scenario 8

Exit Flow Step 1: Reasoning

A modal appears when the candidate clicks "I don't have this document" or "I don't have preferred documents." Step 1 collects the reason, optional details, and an expected date for the document.

Triggered by: Clicking "I don't have this document" or "I don't have preferred documents"
What happens next: Click "Next" to go to Step 2 (Scenario 9)
1
2
3
4
5
6
7

Spec Breakdown

  1. 1
    Modal header
    "Preferred documents not available" title with a close (x) button. Closing the modal discards any unsaved changes and returns to the form.
  2. 2
    Stepper
    Two-step stepper: Step 1 "Reasoning" (active, blue circle) connected by a dashed line to Step 2 "Add alternate documents" (inactive, grey circle). Visually communicates the two-part process.
    Key states: Active step (blue circle + blue text), Completed step (green check), Upcoming step (grey circle + grey text). Dashed connector between steps.
  3. 3
    Note box
    Blue informational box explaining that the preferred document is the most reliable form of verification. Warns that using alternate documents may cause delays in the verification process.
  4. 4
    Reason dropdown (required)
    "Why do you not have the documents?" — a required dropdown (red asterisk). Pre-defined options: Lost, Serving notice period, Company shut down, Never issued, Other.
    Validation: Required field. The "Next" button is disabled until a reason is selected.
  5. 5
    Detail textarea (optional)
    "Explain the reason in detail" with hint/placeholder text. This is an optional field that allows the candidate to provide additional context about why they don't have the document.
  6. 6
    ETA date picker with validation error
    "When can you expect the documents?" date picker field. In this scenario, the candidate has selected a past date, triggering a red border and error message: "ETA cannot be in past".
    Key states: Empty (default), Valid date (green/normal border), Past date error (red border + error message "ETA cannot be in past")
  7. 7
    Modal footer
    Cancel (text/ghost button) on the left and Next (blue filled button) on the right. Cancel closes the modal. Next advances to Step 2 (Scenario 9).
    Key states: Next disabled (reason not selected or validation error), Next enabled (valid form data)
Scenario 9

Exit Flow Step 2: Alternate Docs

Step 1 is completed (green check). Step 2 allows the candidate to upload alternate documents that can be used instead of the preferred document.

Triggered by: Clicking "Next" on Step 1
What happens next: Upload alternate docs, then Submit to close the modal
1
2
3
4

Spec Breakdown

  1. 1
    Stepper
    Step 1 "Reasoning" now shows a green check (completed). Step 2 "Add alternate documents" is active (blue circle). The dashed connector reflects the progression.
  2. 2
    Alternate documents list
    A list of acceptable alternate documents. Each document has its name and an upload button or an uploaded status indicator. The list is determined by the BGV Form Config API based on the section type.
  3. 3
    Uploaded alternate doc
    Shows a file preview row for an already-uploaded alternate document. Includes file name, size, timestamp, and delete option. Confirms the alternate doc is ready.
  4. 4
    Submit button
    Closes the modal and saves the exit flow data (reason, details, ETA, and uploaded alternate documents) to the backend. The form section is then marked as "completed with alternate documents."
    Key behavior: On submit, the sub-section updates to show that alternate docs were provided. The Proceed button in the sub-section becomes enabled.

Non-Preferred First Warning

The candidate has uploaded a non-preferred document first (e.g., Offer Letter instead of the preferred Experience Letter). A warning banner prompts them to upload the preferred document.

Triggered by: Candidate uploads a non-preferred document (e.g., Offer Letter instead of Experience Letter)
What happens next: Upload the preferred doc, or proceed with non-preferred
1
2
3
4

Spec Breakdown

  1. 1
    Warning banner
    Yellow/orange warning banner informing the candidate that the uploaded document is not the preferred one. Prompts the candidate to upload the preferred document (Experience Letter) for faster verification.
    Visual: Amber/yellow background, warning icon. Not dismissible — persists until the preferred document is uploaded.
  2. 2
    Uploaded non-preferred file
    File row showing the Offer Letter that was uploaded. Displays file name, size, timestamp, and delete button. The file is accepted but not ideal.
  3. 3
    Preferred doc still needed
    Visual indication that the Experience Letter (preferred document) is still required or strongly recommended. The upload zone remains available for the preferred document.
  4. 4
    Proceed button
    May be disabled or enabled depending on the configuration. If the preferred document is mandatory, Proceed remains disabled until it is uploaded. If the preferred document is only recommended (not mandatory), Proceed may be enabled with the non-preferred upload.
    Key behavior: Depends on mandatory vs recommended configuration from the BGV Form Config API.
Scenario 11

Mandatory Doc: Reason & ETA (Inline)

The candidate clicks "I don't have this" on a mandatory document. Instead of a modal, an inline form appears within the subsection card for the candidate to provide a reason and expected date.

Triggered by: Candidate clicks "I don't have this" on a mandatory document (inline flow, not modal)
What happens next: Fill reason + ETA, then proceed
1
2
3

Spec Breakdown

  1. 1
    Reason dropdown (inline)
    Same dropdown options as the Exit Flow modal (Lost, Serving notice period, Company shut down, Never issued, Other) but displayed inline within the subsection card rather than in a separate modal. Required field.
    Key difference from Scenarios 8-9: This is an inline flow — no modal is opened. The fields appear directly in the card, below or replacing the upload zone.
  2. 2
    ETA date field (inline)
    Date picker for when the candidate expects to have the document. Same validation as the modal version — past dates trigger an error. Displayed inline in the subsection card.
    Validation: Cannot select past dates. Error message: "ETA cannot be in past"
  3. 3
    Expanded inline form
    The reason and ETA fields appear directly inside the subsection card, replacing or appearing below the upload zone. This inline approach is used for mandatory documents where the exit flow is simpler (no alternate documents step).
    Key behavior: After filling reason + ETA, the Proceed button becomes enabled. The sub-section is marked as "completed with reason" rather than "completed with document."
Scenario 12

Review & Submit

All sections are completed (or marked with reasons). The candidate reviews a summary of everything they have submitted and provides consent before final submission.

Triggered by: All sections completed (or marked with reasons)
What happens next: Submit the form
1
2
3
4
5

Spec Breakdown

  1. 1
    Sidebar "Final Step"
    "Review & Submit" item in the sidebar, marked as the final step. All previous sections (Identity, Education, Address, Employment) show green checks indicating completion.
  2. 2
    Section summary cards
    Each section (Identity, Education, Address, Employment) is displayed as a summary card showing: uploaded documents with their names, status indicators (uploaded, alternate provided, reason given), and "Edit" links that take the candidate back to that section to make changes.
    Key behavior: Clicking "Edit" navigates back to the specific section. Changes made there are reflected in the review summary when the candidate returns.
  3. 3
    Missing doc warning
    Yellow warning banner for any section where a mandatory document was not uploaded. Shows the reason and ETA provided by the candidate (from the Exit Flow or inline form). This is informational — it does not block submission.
    Visual: Amber/yellow background with warning icon. Displays: document name, reason text, ETA date.
  4. 4
    Consent checkbox
    Required checkbox the candidate must check before submitting. Confirms that all information provided is accurate and truthful. The Submit button remains disabled until this checkbox is checked.
    Key states: Unchecked (Submit disabled), Checked (Submit enabled)
  5. 5
    Submit button
    Green "Submit Background Verification Form" button. Full width, prominent styling. This is the final action that submits the entire form.
    Key states: Disabled (consent not checked — opacity 0.4), Enabled (consent checked — full green, clickable), Submitting (loading spinner, disabled). On success: redirect to confirmation page.
